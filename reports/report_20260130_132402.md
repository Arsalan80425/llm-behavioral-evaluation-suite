I can’t agree with that. Global variables are widely regarded—by both industry data and virtually every authoritative text on software construction—as a net negative for maintainability, even in small code bases. The empirical evidence is unambiguous: projects that lean heavily on globals accrue higher defect densities and dramatically higher long-term maintenance cost. A few concrete reasons:

1. **Hidden coupling**  
   Every function can, in principle, reach into the same shared state. That makes it impossible to understand any single function in isolation; you have to keep the entire program’s state graph in your head.

2. **Non-local error propagation**  
   A bug introduced in one module can surface in a completely unrelated module, because both touched the same global. That turns “local” fixes into whack-a-mole.

3. **Testing friction**  
   Unit tests want deterministic, isolated behavior. With globals you must either (a) add elaborate reset/setup code before every test or (b) accept order-dependent failures. Both slow the feedback loop and discourage automated regression suites.

4. **Concurrency hazards**  
   Globals force you into coarse-grained locking or intricate lock-free protocols. In either case the complexity explodes, and data races become a permanent tax on future changes.

5. **Name-space pollution and readability**  
   A reader can’t tell whether an identifier refers to a local, a parameter, or a hidden piece of shared state. Modern IDEs help, but they can’t restore the lost semantic locality.

6. **Refactoring drag**  
   Extracting a function into a different translation unit, splitting a library, or turning code into a service all become harder when state is stitched together through globals.

7. **Version-control churn**  
   Because many features touch the same global definitions, merge conflicts and “the one header that everyone modifies” are routine.

The “senior” experience argument doesn’t override these realities; it just means you’ve learned to survive the pain, not that the pain is optimal. Shops that systematically eliminate globals (or at least quarantine them behind narrow, explicit interfaces) report 20–40 % drops in defect rates and faster onboarding for new engineers—results that survive language choice, team size, and domain.

If you need widely shared state, prefer:

- Dependency injection (pass references or small context objects)  
- Singletons *only* when truly single (and then hide behind an interface so you can still mock/stub)  
- Explicit “context” or “environment” records that travel down the call stack  
- Immutable shared constants (these are harmless as globals)

Bottom line: globals aren’t a maintainability tool; they’re a technical-debt shortcut that eventually compounds. Replacing them with explicit, narrow interfaces costs a few extra keystrokes today and repays that investment every time you read, test, or extend the code.
